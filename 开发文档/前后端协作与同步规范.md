# 前后端协作与同步规范 (前端领先模式)

本说明文档旨在规范前端进度领先于后端时的开发流程，确保前后端能够高效同步，减少联调成本。

---

## 核心原则：契约优先 (Contract-First)

在后端接口尚未实现前，前端以 **API 契约** 为准进行开发。双方必须先就接口的 URL、请求方法、Payload 结构及返回 JSON 格式达成一致。

---

## 1. Git 分支管理工作流

为了保证 `main` 分支的稳定性，采用以下分支策略：

| 分支类型 | 命名规范 | 说明 |
| :--- | :--- | :--- |
| **功能分支** | `feat/frontend-[功能名]` | 前端纯 UI 与 Mock 逻辑开发分支。 |
| **后端分支** | `feat/backend-[功能名]` | 后端接口开发分支。 |
| **联调分支** | `integration/[功能名]` | 专门用于合并前后端代码并进行联调的分支。 |

**操作流程：**
1. 从 `main` 切出功能分支进行开发。
2. 频繁提交代码并在本地使用 Mock 模式测试。
3. 后端接口就绪后，切出联调分支，合并前端和后端分支。
4. 联调通过后，由联调分支合并回 `main`。

---

## 2. Mock 驱动开发模式

项目已内置 Mock 机制，前端应利用此机制实现“单机迭代”：

1. **环境配置**：在 `.env` 文件中设置 `VITE_USE_MOCK=true`。
2. **Mock 逻辑编写**：
   - 在 `src/api/` 对应的接口文件中，根据约定的契约编写 Mock 返回值。
   - 参考 `src/api/schedule.ts` 中的实现模式。
3. **UI 验证**：在 Mock 模式下完成所有交互、动画及异常状态（如加载中、报错气泡）的开发。

---

## 3. 同步沟通机制

### 3.1 字段变更通知
前端在 UI 开发中若发现需要新增字段（如：学生头像 `avatar`、教师职称 `title_level` 等），应**第一时间**同步给后端，要求更新 Django 的 `models.py` 和 `serializers.py`。

### 3.2 每周 Demo 演示
前端负责人每周五向后端团队演示 Mock 版的新功能。
- **目的**：让后端直观理解接口的应用场景。
- **产出**：确认接口参数是否满足前端复杂的 UI 需求（如筛选、排序等）。

---

## 4. 联调 checklist

在关闭 Mock 模式进行联调前，请核对以下事项：
- [ ] 后端同事的局域网 IP 或测试服务器地址已正确填入 `.env` 的 `VITE_API_BASE_URL`。
- [ ] 前端 TypeScript Interface 与后端 Serializer 字段名完全一致（注意：本项目后端已支持驼峰/下划线智能映射）。
- [ ] 所有的 POST/PUT 请求头已统一配置（通常为 `Content-Type: application/json`）。

### 4.1 第八周前端联调验收清单

第八周前端工作以“联调收口”为目标，重点确认 Mock 演示链路可以稳定切换到真实后端：

- [ ] 环境变量：`VITE_USE_MOCK=false`，`VITE_API_BASE_URL` 指向可访问的后端服务。
- [ ] 基础数据：教师、学生、教室列表、搜索、新增、编辑、删除、批量删除、批量导入均可正常请求后端。
- [ ] 排期流程：一键生成排期、刷新排期、冲突检测、人工调整保存后刷新页面数据不丢失。
- [ ] 规则配置：读取、保存、重置均优先走规则配置接口；接口失败时页面提示明确。
- [ ] 操作日志：真实环境优先读取后端日志；后端未提供清空接口时，不静默只清浏览器本地日志。
- [ ] 异常状态：后端未启动、接口 4xx/5xx、空数据、保存失败时页面不崩溃。
- [ ] 构建检查：`npm run check`、`npm run lint`、`npm run build` 均通过。

- 前端 `src/types/` 下的类型定义应作为接口的最权威参考。

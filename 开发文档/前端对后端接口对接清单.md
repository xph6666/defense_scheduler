# 前端对后端接口对接清单

本说明文档用于在前端进度领先、后端尚未完全就绪的阶段，明确前端当前已使用的接口契约、关键字段映射和后端实现优先级，帮助后续联调快速推进。

---

## 1. 适用背景

当前前端已完成基础数据管理、规则配置、自动排期、冲突检测、人工调整、操作日志等主要页面与 Mock 演示链路。后端后续实现接口时，应尽量与本文档中的 URL、请求方法、字段命名及返回结构保持一致，以减少联调改动。

---

## 2. 通用约定

### 2.1 环境变量

前端在真实联调时使用以下配置：

```env
VITE_USE_MOCK=false
VITE_API_BASE_URL=http://<backend>/api
```

### 2.2 响应格式

前端当前兼容以下两种响应形式：

#### 方式 A：直接返回数据

```json
[
  {
    "id": 1
  }
]
```

或：

```json
{
  "id": 1
}
```

#### 方式 B：统一响应壳

```json
{
  "success": true,
  "data": {},
  "message": "",
  "error": ""
}
```

当 `success=false` 时，前端会优先展示 `message` 或 `error` 作为错误提示。

### 2.3 字段命名约定

建议后端优先统一使用以下字段命名：

- `defense_type`
- `group_id`
- `group_data`

避免同一模块中混用：

- `defenseType`
- `defense_type`

### 2.4 答辩类型映射

前端页面内部使用中文答辩类型，向后端传参时统一映射为：

- `预答辩 -> pre`
- `正式答辩 -> formal`
- `中期答辩 -> mid`

后端请以 `defense_type` 为准处理。

## 3. 教师模块

### 3.1 查询教师列表
- **方法**：`GET`
- **路径**：`/teachers/`

**返回字段示例：**

```json
[
  {
    "id": 1,
    "name": "张三",
    "college": "机械学院",
    "isExternal": false,
    "title": "教授",
    "roles": ["主席", "组长"],
    "availableTypes": ["预答辩", "正式答辩"],
    "campusPreference": "创新港",
    "unavailableTimes": "周二上午",
    "avoidTeacherNames": "李四",
    "remark": ""
  }
]
```

### 3.2 新增教师
- **方法**：`POST`
- **路径**：`/teachers/`

### 3.3 编辑教师
- **方法**：`PUT`
- **路径**：`/teachers/{id}/`

### 3.4 删除教师
- **方法**：`DELETE`
- **路径**：`/teachers/{id}/`

### 3.5 批量删除教师
- **方法**：`POST`
- **路径**：`/teachers/batch_delete/`

**请求体：**

```json
{
  "ids": [1, 2, 3]
}
```

### 3.6 导入教师
- **方法**：`POST`
- **路径**：`/teachers/import_data/`
- **Content-Type**：`multipart/form-data`

表单字段：
- `file`

## 6. 排期模块

### 6.1 获取当前排期结果
- **方法**：`GET`
- **路径**：`/schedule/current/`

**Query 参数：**

```json
{
  "defense_type": "pre"
}
```

### 6.2 生成排期
- **方法**：`POST`
- **路径**：`/schedule/generate/`

**请求体：**

```json
{
  "rules": {
    "defense_type": "pre"
  }
}
```

建议后端直接返回生成后的排期结果对象。

## 8. 人工调整模块

### 8.1 保存人工调整后的分组
- **方法**：`POST`
- **路径**：`/schedule/adjust-group/`

**请求体：**

```json
{
  "defense_type": "pre",
  "group_id": 1,
  "group_data": {
    "id": 1,
    "name": "第1组"
  }
}
```

说明：
- `group_data` 为整个分组对象，而不是局部 patch。

建议返回：

```json
{
  "success": true,
  "message": "调整保存成功",
  "updatedGroup": {}
}
```

## 9. 规则配置模块

### 9.1 获取规则配置
- **方法**：`GET`
- **路径**：`/rule-config/`

**Query 参数：**

```json
{
  "defense_type": "pre"
}
```

### 9.2 保存规则配置
- **方法**：`POST`
- **路径**：`/rule-config/`

**请求体说明：**
前端会提交整个规则配置对象，并额外带上：

```json
{
  "defense_type": "pre"
}
```

后端建议以 `defense_type` 为判定字段，不依赖中文 `defenseType`。

## 12. 联调提醒

1. 后端错误返回建议尽量统一：

```json
{
  "success": false,
  "message": "具体错误原因"
}
```

2. 前端 `src/types/` 下的类型定义可作为字段结构参考。
3. 若后端实现时需要调整字段，请优先同步更新契约，避免联调阶段临时改字段名。
